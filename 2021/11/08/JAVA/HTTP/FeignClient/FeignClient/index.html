FeignClient

Fegin 은 Netflix 에서 개발된 REST Call 을 추상화 한 Spring Cloud Netflix 입니다.

HTTP 통신할려고 하는 HTTP URL 또는 Application name 에 매핑 해서 직관적으로 명시해 통신 할수 있습니다.

본격적으로 사용 방법에 대해 알아보겠습니다.

라이브버리 추가

@EnableFeignClients 추가하기

@EnableFeignClients

만약 @EnableFeignClients 추가가 안되었다면 FeignClient 를 사용하기 위해 추가 하도록 합니다.

internal url 설정하기

application.yml 파일에 통신 할려고 하는 서버 url 를 설정 합니다.

Client Interface 생성하기

http://localhost:8080/api/boards 호출 합니다.

Request 요청 중 Pagination Parameter 는 offset, limit 가 있습니다.

@RequestParam 을 추가해 바인딩 합니다.

여기서 name 은 discovery service(netflix eureka) 사용하는 경우 해당되는 서비스 이름 입니다.

아직 사용하지 않고 있으니 적절하게 name 을 만들어서 사용합니다.

Controller, Server 생성하기


Service 단에는 Client Interface 생성한 BoardServiceClient 주입 합니다.

주입한 BoardServiceClient 에서 getBoardV1 메소드를 실행 합니다.

호출 결과가 정상적으로 응답 받았습니다.


요청 파라미터를 POJO 또는 Map 매개변수로 사용하는 방법 입니다. 요청 파라미터 전용 객체(BoardRequest)를 생성해서 요청 합니다.

Spring Cloud OpenFeign 에서는 @SpringQueryMap 을 제공 해서 편리하게 요청 파라미터 전용 객체 통해 요청 할 수 있습니다.

ErrorDecoder 이용해서 예외 핸드러 처리 해보자

마이크로 서비스의 서비스간 통신을 할 경우 특정 원인으로 요청 할려는 서버로 부터 4xx, 5xx 에러 응답을 받을수 있습니다.

이때 우리는 적절한 예외처리를 해줘야 합니다. try catch 로 적절하게 대응 할 수 있지만 코드상 지저분해지기 때문에 ErrorDecoder 이용해서 예외처리를 해볼려고 합니다.

기존에 요청 ‘api/boards’ 를 ‘api/boardsTEST’ 로 변경해서 일부러 404 에러를 발생 하도록 합니다.

에러 발생시 FeignErrorDecoder 는 에러 핸드링 통해 예외처리를 할수 있도록 합니다.

header 값을 추가해 요청 해보자

게시판 등록 할려고 합니다.

Header Key 값은 ‘accessToken’ 입니다. Value 값을 통해 인증 처리 하게 됩니다.

Circuitbreaker 적용 해보기

Scheduler Server 로부터 Server Api 로 호출 한다고 가정했을때 만약 Server Api 가 특정 문제로 서버가 중단 되는 경우

요청한 Scheduler Server 는 Time Out 에러가 발생 하게 됩니다.

이 경우 연쇄적으로 Scheduler Server 도 에러가 발생되는데요. 이후 로직도 실행이 멈추게 되고 전체적인 서비스 또한 멈추게 됩니다.

하나의 서버가 중단으로 인해 여러 서버까지 피해 입게 됩니다.

Server Api 서버가 복구 될때까지 Scheduler Server 는 Server Api 호출을 잠시 멈추고 이후 로직을 실행해야 하고 이후 Server Api 복구 된다면

요청을 시작 할수 있도록 하는것이 바람직한 방향 일겁니다.

이러한 방향을 도와주는것이 Circuitbreaker 기능 입니다.

즉 해당 서버가 중단 되었으면 반복적으로 서버 호출을 멈추게 하고 다시 서버가 회복 되었으면 원래 흐름대로 호출 할 수 있도록 하는 것 입니다.

Spring Boot 2.4.0 이전 버전 + Spring Cloud 2020.0.0 버전 이전에는 netflix-hystrix 사용하면 되었지만 이후 버전에는 resilience4j 사용 합니다. (hystrix 는 Deprecated 되었습니다.)

여기서는 resilience4j 로 해보겠습니다.

resilience4j 라이브러리 추가 합니다.

Resilience4J Default 로 제공하는 설정값을 변경하고자 할때 Customizer Custom 해서 설정값을 변경 할 수 있습니다.

.failureRateThreshold(8)

해당 설정은 호출 실패 확률 퍼센트 설정 값입니다. 8으로 입력시 100 번 호출 중 8번만 호출 실패시 Circuitbreaker 작동해서 해당 호출을 멈추게 됩니다.

Default 값은 50 입니다.

.waitDurationInOpenState(Duration.ofMillis(1000)) // 1초

해당 설정은 Circuitbreaker 작동해서 해당 호출을 멈추게 될때 언제까지 멈추게 할것인가 입니다.

Default 값은 60초 입니다.

.slidingWindowType(CircuitBreakerConfig.SlidingWindowType.COUNT_BASED)

해당 설정은 Circuitbreaker 작동해서 해당 호출을 멈추게 되고 이후 서버가 정상적으로 작동할때 Circuitbreaker 는 멈추게 되는데 그 사이에 호출된 카운트가 몇번 인지

기록 하는 설정값입니다. 참고로 카운트 기반이 아니라 시간으로 변경하고 싶으면

.slidingWindowType(CircuitBreakerConfig.SlidingWindowType.TIME_BASED)

로 설정 하면 됩니다.

.slidingWindowSize(2)

해당 설정은 Circuitbreaker 작동이 멈출때 호출 결과 기록하는데 있어서 sliding window 사이즈 값 입니다.

.timeoutDuration(Duration.ofSeconds(3))

해당 설정은 만약 호출 할려고 하는 서버가 3초동안 응답이 없을 경우 Circuitbreaker 작동 하겠다는 의미 입니다.

우선 사용하고자 하는 Service 단에 CircuitBreakerFactory 주입 합니다.

circuitBreaker 생성 하기 위해 circuitBreakerFactory 이용해 create 합니다.

생성한 circuitBreaker 을 run 메소드 실행 하면 됩니다. 만약 문제가 발생하면 null 로 치환하도록 합니다.

호출해야 하는 서버를 중단시키고 실행 하도록 합니다. 원하는대로 null 로 반납하게 됩니다.
