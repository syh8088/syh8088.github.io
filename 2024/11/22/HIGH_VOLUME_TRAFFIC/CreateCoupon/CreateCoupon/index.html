대용량 트래픽 - 선착순 쿠폰 등록

특정 시간에 쿠폰을 등록 할 수 있도록 해서 사용자가 웹페이지 상에 쿠폰을 등록 하도록 기능 개발 했다고 가정 하겠습니다.

쿠폰은 200개 만 등록 할 수 있도록 설정 되어 있습니다. 특정 시간에 많은 사용자가 동시 다발적으로 쿠폰을 등록시 서비스 상 문제 없도록 개발 해야 합니다.

200개 이상 쿠폰 등록 된 경우 이후 사용자는 아쉽지만 쿠폰 등록은 할 수 없도록 개발 해야 합니다.

선착순 쿠폰 등록 개발하기

CouponService 에서 createCoupon 메소드를 확인하면 우선 등록된 쿠폰 카운트를 조회 하도록 합니다.

조회 후 ‘LIMITED_NUMBER_COUPONS’ 보다 작으면 즉 등록된 쿠폰 등록 제한 200 이하이면 userId 값을 전달해 쿠폰 등록을 합니다.

만약 등록된 쿠폰 갯수가 200 이상이면 더 이상 쿠폰을 등록 하지 않도록 해야 합니다.

대규모 트래픽 TEST 를 할 수 있도록 ExecutorService 객체를 이용해서 1000 스레드를 한꺼번에 생성해서 바로 쿠폰 등록을 하도록 하겠습니다.

이상적인 시나리오 이라면 선착순 200 횟수가 지나면 이 후 스레드는 더 이상 쿠폰 등록을 할 수 없도록 해야 합니다.

TEST 결과 등록된 쿠폰 카운트가 200개를 기대했었는데 889개를 등록 된것으로 확인 할 수 있습니다.

왜 이런 결과가 일어 나는 걸까요?

설명을 쉽게 할 수 있도록 1번 스레드와 2번 스레드로만 설명 하겠습니다. 우선 1번 스레드가 등록된 쿠폰 카운트를 조회 하고 곧 바로 2번 스레드도

등록된 쿠폰 카운트를 조회 합니다. 각각 스레드는 조회된 등록된 쿠폰 횟수가 199 개 이기 때문에 1번 및 2번 스레드는 모두 쿠폰을 등록 할 수 있게 됩니다.

원래 둘 중 하나 스레드만 등록이 가능 해야 하는데 두개의 스레드가 공유 데이터 접근 및 동시적으로 작업으로 인해 Race Condition 이 발생 된던 것 입니다.

Redis INCR 명령어 통해 문제를 해결 해보자

우선 Redis 의 특징을 알아보자면 싱글 스레드 기반으로 동작 합니다. 이런 싱글 스레드로 인해 뛰어난 성능을 자랑하고 개발자가 잘 이용하면 사이드 이펙트가

없다는게 큰 장점 입니다.

이러한 특징 덕분에 다수의 스레드가 동일한 자원에 접근시 동시성 문제를 해결 할 수 있도록 합니다.


우선 특정 스레드가 쿠폰 등록시 redis 의 INCR ${key} 기능을 이용해서 명령어 조회수 1씩 증가 하도록 합니다. 이때 싱글 스레드 기반으로 동작하기 때문에
동시성 문제는 발생하지 않습니다.

이때 등록된 쿠폰 횟수가 200 이하 이라면 쿠폰 등록을 할 수 있도록 하고 등록된 쿠폰 횟수가 200 이상이라면 쿠폰 등록을 할 수 없도록 구성 합니다.

CreateCouponCountRedisRepository 이라는 Redis 전용 Repository 클래스를 만듭니다. ‘coupon_count’ 이라는 Redis Key 값을 활용해서

increment 메소드를 호출시 Redis 명령어 ‘INCR coupon_count’ 실행 하도록 합니다.

등록된 ‘CreateCouponCountRedisRepository’ 클래스를 참조해서 쿠폰 등록 전 ‘createCouponCountRedisRepository.increment();’ 실행해

등록된 쿠폰 횟수를 조회 하도록 합니다. 이때 200 이하 이라면 정상적으로 쿠폰 등록 하도록 합니다.

이렇게 수정 후 다시 TEST 해보면 성공적으로 쿠폰 등록 된것을 확인 할 수 있습니다.

nGrinder 통해 성능 분석 하기

이번에는 nGrinder 통해 성능 분석을 해보겠습니다. 효과적인 성능 분석을 위해 쿠폰을 등록하는 비지니스 로직을 약 3초 걸린다고 가정하게 생각해보겠습니다.

일부러 Thread.sleep 을 약 3초 지연 실행해서 지연 해보도록 하겠습니다.

평균 TPS 가 1,133.3 정도 나왔는데요. 문제는 그래프에서 초기 단계를 보면 처리량이 높지 않다가 이후 급속도록 높아지는 것을 확인 할 수 있습니다.

이것은 먼저 진입한 초기 많은 스레드가 쿠폰 등록시 처리를 완료하기 위해 시간이 지연되고 이 후에 진입한 스레드가 처리가 완료 될때까지 대기 하는 문제가 있기 때문 입니다.

즉 실제로 사용자는 늦게 쿠폰 등록한 사용자라면 대기 상태에 빠져 버리게 됩니다.

물론 Thread Pool 를 증가 시켜 이를 문제를 해결 할 수 있습니다. 지금 TEST 에서는 Thread.sleep 로 통해 시간 지연을 했을 뿐이지 실제로는 디비 서버와

많은 I/O 작업을 일어나고 전반적인 성능 저하로 이어지게 되어서 100% 문제 해결이라고 볼 수 없을 것 입니다.

Kafka 로 통해 문제 해결하기

쿠폰 등록하는 무거운 작업들을 Kafka 를 활용해서 messageQ를 적용 해보기로 했습니다. 대용량 트래픽에 안정성을 확보하기 위함 인데요.

쿠폰 등록하는 부분을 일단 Kafka 에 메시지 큐로 등록해서 후속 처리 하도록 하고 메시지 등록만 완료된다면 바로 클라이언트 사용자에게 응답을 할 수 있도록 합니다.

그럼 이전 보다 더 빠른 처리량을 확보 할 수 있게 됩니다.

우선 zookeeper 를 실행 하도록 합니다.

마지막으로 Kafka 를 실행 하도록 합니다.

‘create-coupon’ 이라는 topic 을 생성 하도록 합니다.

CreateCouponKafkaProducer 클래스를 생성해서 ‘create-coupon’ 이라는 Topic 이름에 Producer 를 생성 하도록 합니다.

쿠폰 등록시 이전에는 복잡한 로직인 쿠폰 등록 로직을 실행 했었는데요. 이번에는 등록한 CreateCouponKafkaProducer 에

create 메소드를 실행 해서 카프카에 메시지 등록을 할 수 있도록 합니다.

Consumer 서버단에서는 ‘create-coupon’ 이라는 topic 이름으로 KafkaListener 등록 합니다.

이전과 비교해서 초기 진입한 많은 스레드가 Kafka 에 쿠폰 메시지 등록 이후 바로 응답 해주고 있기 때문에 초기 부터 높은 TPS 를 확인 할 수 있습니다.
