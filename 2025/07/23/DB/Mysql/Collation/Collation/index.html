Collation

Collation 는 문자열을 비교하고 정렬할 때 적용되는 규칙 (정렬 기준) 입니다. 즉 “문자 집합(Character Set)”을 사용하는 문자열이라도, 어떤 Collation 을 적용하느냐에 따라

  • 정렬 순서 (ORDER BY)
  • 비교 (=, LIKE, > 등)
  • 대소문자 구분 여부
  • 악센트 구분 여부

에 따라 모두가 달라질 수 있습니다.

MySQL 에서 Collation 네이밍 컨벤션

// 이미지

Mysql 에서 Collation 이름은 크게 네 부분으로 구성 되어 있습니다. 각각 Collation 중요한 특징이 있으며 하나하나씩 알아보도록 하겠습니다.

문자 집합 (Character Set)

MySQL 에서 Collation (정렬 규칙) 은 문자 집합 (Character Set) 에 종속 되어 있습니다. 즉 Collation 은 어떤 문자 집합(Character Set)에 대해 적용되는 문자 정렬 및 비교 규칙입니다.

예시로 가장 많이 사용되어 있는 문자 집합 utf8mb4 로 설명 하도록 하겠습니다.

문자 집합 (utf8mb4) Collation 종류 설명
utf8mb4 utf8mb4_general_ci 빠르지만 비교 정확도 낮음
utf8mb4 utf8mb4_unicode_ci 유니코드 표준에 따른 정렬
utf8mb4 utf8mb4_bin 바이트 값 기준 정렬 (대소문자/악센트 구분 O)

Collation utf8mb4_general_ci, utf8mb4_unicode_ci, utf8mb4_bin 는 모두 문자 집합 utf8mb4 에 종속 되어 있습니다. 모두 같은 문자 집합 (utf8mb4) 이지만 Collation 만 다르면 정렬과 비교 결과가 달라집니다.

문자 집합 (Character Set) 경우는 데이터를 어떻게 등재 할 것인가 예를들어 유니코드 기반 저장소 로 저장 할 것인가 결정 되는 부분이고 Collation 는 등재 된 데이터를 어떻게 정렬 할 것이고 비교 할 것인가 방법을 나타내는 것 입니다. (예: 가나다순? ABC순? 악센트 구분할까?)

언어 종속 (Language-specific)

문자열을 비교하거나 정렬할 때 그 언어의 규칙(철자 순서, 대소문자 구분, 악센트 처리 등)을 따르는 Collation 이 있다는 것 입니다. 예를들어

이렇게 ㄱ->ㄴ->ㄷ->ㄹ 한글 사전 순 정렬 순으로 따르게 됩니다.

UCA 버전

UCA (Unicode Collation Algorithm) 는 유니코드에서 정의한 국제 표준 정렬 알고리즘 입니다. 즉 전 세계 모든 언어의 문자들을 “사전순으로 어떻게 정렬할 것인가” 를 유니코드에서 통일된 방식으로 정의한 규칙 입니다.

UCA 버전 이란 특정 Collation이 어느 버전의 UCA(Unicode Collation Algorithm)를 기준으로 만들어 졌는지를 나타내는 것입니다. 유니코드(Unicode)는 버전이 올라 가면 올라 갈수록 더 많은 문자와 정렬 규칙이 추가 되는데 이것에 대응 해서 UCA 버전 도 올라가면서 훨씬 정확하고 최신 유니코드 정렬 할 수 있도록 해줍니다.

UCA 버전 대응 유니코드 버전 특징
UCA 4.0 Unicode 4.0 MySQL 5.x 기본
UCA 5.2 Unicode 5.2 일부 Collation 향상
UCA 9.0 Unicode 9.0 MySQL 8.0에서 새롭게 도입됨
UCA 14.0+ UCA 14.0+ 최신 정렬 및 문자 반영 (예: 새로운 이모지)

MySQL 에서 UCA 버전 중 0900 하고 general 차이

Collation 정렬 결과
utf8mb4_general_ci A, Z, a, á, Ä, æ, 가, 각, 간, 😀, 😃, 😂 (이모지 무시 및 뒤로 밀림)
utf8mb4_0900_ai_ci A, a, á, Ä, æ, Z, 가, 각, 간, 😂, 😃, 😀
utf8mb4_0900_as_cs A, Z, a, á, Ä, æ, 가, 각, 간, 😂, 😃, 😀
utf8mb4_general_as_cs A, Z, a, á, Ä, æ, 가, 각, 간, 😀, 😃, 😂 (이모지 무시 및 뒤로 밀림)

0900 하고 general 가장 큰 차이는 정확한 유니코드 사전 순 정렬 입니다. 단순한 알파벳 및 악센트 문자으로만 보면 정렬 순서는 거의 차이가 없다고 볼 수 있겠지만 더 넓은 문자 범위 (이모지, 다국어, 악센트 민감도, case 구분 등) 로 가면 UCA 9.0 기반인 0900 이 더 정확한 유니코드 정렬 기준을 보장 합니다.

민감도

민감도는 문자열을 비교할 때 특정 요소(대소문자, 악센트 등)를 구분할지 무시할지를 결정하는 특성 입니다.

민감도의 종류

민감도 이름 접미어 의미 예시 (=, ORDER BY) 설명
Case Sensitivity (CS) _cs 대소문자 구분함 ‘A’ != ‘a’ A와 a는 다름
Case Insensitivity (CI) _ci 대소문자 무시함 ‘A’ = ‘a’ A와 a는 같음
Accent Sensitivity (AS) _as 악센트 구분함 ‘a’ != ‘á’ a와 á는 다름
Accent Insensitivity (AI) _ai 악센트 무시함 ‘a’ = ‘á’ a와 á는 같음
Kana Sensitivity (일본어) _ks 히라가나/가타카나 구분 ‘あ’ != ‘ア’ 일본어 전용
Width Sensitivity _ws 전각/반각 구분 ‘A’ != ‘Ａ’ 일본어, 중국어 등에서

민감도의 선택 기준

상황 추천 민감도
사용자 이름 및 일반 검색 ai_ci (악센트/대소문자 무시)
비밀번호 및 코드 비교 as_cs (정확한 비교 필요)
국제화 및 다국어 정렬 as_cs 또는 unicode_ci
이모지 및 특수 문자 구분 bin (바이트 단위 비교)

Collation 동작 방식

우선 문자열 데이터가 어떻게 저장되는지 부터 알아보도록 하겠습니다. 유니코드 기반 문자 집합에서는 문자 데이터를 저장할 때 그 문자에 해당하는 유니코드 코드 포인트 값을 인코딩해서 저장합니다.

UTF-8 인코딩 방식은 가변 길이 인코딩 방식으로 코드 포인트 범위에 따라 문자가 1 ~ 4바이트로 인코딩 됩니다.

/// 실제 UTF-8 인코딩 과정 이미지

코드 포인트 범위 사용 비트수 인코딩 구조 (인코딩 포맷) 인코딩 바이트
U+007F 7 0xXXXXXX 1Byte
U+0080 ~ U+07FF 11 110xxxxx 10XXXXXX 2Byte
U+0800 ~ U+FFFF 16 1110xxxx 10xxxxxx 10XXXXXX 3Byte
U+10000 ~ U+1FFFF 21 11110xxx 10xxxxxx 10xxxxxX 10XXXXXX 4Byte

이 표는 실제 UTF-8 인코딩 과정을 돕기 위한 UTF-8 코드 포인트 범위, 사용 비트수, 인코딩 구조 (인코딩 포맷), 인코딩 바이트 를 나타낸 것 입니다. 코드 포인트 범위 별로 코드 포인트를 표현하는데 사용되는 비트 수가 다르고 이를 기반으로 UTF-8 인코딩은 지정된 인코딩 구조에 맞추어서 가변 길이의 바이트 시퀀스로 변환하여 저장합니다.

예를들어 문자 가 의 유니코드 코드 포인트는 AC00 으로 UTF-8 인코딩에서 3Byte 필요한 범위에 속합니다. AC00을 비트로 표현하면 1010(A) 1100(C) 0000(0) 0000(0) 값으로 표현이 됩니다.

이를 3Byte 인코딩 구조에 맞춰 값을 배치하면

11101010
10110000
10000000

3Byte 시퀀스 값이 완성됩니다. 이 3Byte 시퀀스는 다시 16진수로 EAB080 으로 표현할 수 있습니다. UTF-8 인코딩으로 최종 EAB080으로 표현되고 저장 하게 됩니다.

DUCET (Default Unicode Collation Element Table)

DUCET (Default Unicode Collation Element Table) 은 유니코드 컨소시엄에서 정의한 문자 정렬 표준 입니다. 전 세계 문자들에 대해 어떻게 정렬하고 비교 할지에 대한 우선순위(가중치) 를 정리한 테이블 입니다.

가중치 종류 예시
Primary 주 가중치 (1차) 문자 간 의미상의 차이 (기본적 구분) a vs b, 가 vs 나
Secondary 보조 가중치 (2차) 악센트나 자모의 차이 e vs é, 한 vs 한̇
Tertiary 3차 가중치 (3차) 대소문자, 글꼴 차이 a vs A, 한 vs 한 (호환형)
Quaternary 4차 가중치 (4차) 구분이 더 필요한 경우 (희귀) 일부 특수문자나 사용자 지정 구분에 사용됨

DUCET (Default Unicode Collation Element Table) 에서는 문자별 가중치 값을 Primary, Secondary, Tertiary와 같이 단계적으로 구성되어 있습니다.

1단계에 해당하는 Primary Weight은 기본이 되는 문자를 비교할 때 사용 되고 2단계인 Secondary Weight은 악센트를 구분할 때, 3단계인 Tertiary Weight 는 대소문자 구분 시 주로 사용 됩니다.

예를들어 민감도 가 ai_ci 경우는 악센트와 대소문자 구분을 하지 않으니 1단계 가중치 값 까지만 사용되고, as_ci 경우는 악센트는 구분하고 대소문자 구분을 하지 않으니 2단계 가중치 값 까지만 사용 됩니다. 마지막으로 as_cs 는 3단계 가중치 값까지 사용 됩니다.

예시로 문자 a 경우 유니코드 코드 포인트 값 0061 로 나타나고 그 뒤로 대괄호 안에 들어있는 3개의 16진수 값이 가중집 값에 해당합니다. 즉 Primary 1C47, Secondary 0020, Tertiary 0002로 정렬 하겠다는 의미 입니다.

MySQL 에서 제공하는 내장 함수(built-in) 함수인 WEIGHT_STRING 이라는 함수를 사용하면 문자열의 가중치 값을 바이너리 형태로 확인할 수 있습니다. 예를들어 문자 a 를 통해 DUCET 의 가중치 값을 확인 하기 위해 HEX, WEIGHT_STRING 함수 통해 가중치 값을 16진수 값으로 확인할 수 있습니다.

utf8mb4_0900_ai_ci 으로 설정 한 다음에 확인 해보면 1단계 즉 Primary 단계에 해당하는 값과 동일한 것을 알 수 있습니다. utf8mb4_0900_as_cs 으로 설정 한 다음에 확인 해보면 가중치 데이터의 세 가지 값에 대해 각각의 값들을 제로바이트로 연결해서 조합한 최종적인 값을 확인 할 수 있습니다.

다음은 collation 테이블을 생성하고 각각 a, b, c 를 INSERT 를 한 다음에 전체 조회를 해서 DUCET 의 가중치 값을 확인 하기 위해 HEX, WEIGHT_STRING 함수 통해 SELECT 해봅니다. 결과 값을 보면 가중치 값 순서대로 데이터가 정렬돼서 반환된 것을 알 수 있습니다.

DUCET (Default Unicode Collation Element Table) 와 UCA (Unicode Collation Algorithm) 차이점은?

DUCET (Default Unicode Collation Element Table) 은 사전으로 생각하면 쉽습니다. 예를들어 a 와 A 는 정렬 대상에서 1순위 즉 더 높은 tertiary 각각의 문자에 어떤 정렬 기준(가중치) 을 줄지 미리 정의된 표 입니다.

반면 UCA (Unicode Collation Algorithm) 는 이 사전을 이용해서 정렬하는 방법 입니다. 즉 문자열을 가지고 어떻게 순서대로 비교 및 정렬 할지 정하는 알고리즘 입니다.

users 테이블 PK user_id 필드 collation 은 utf8mb3_general_ci 로 설정 하였고 user_access 테이블 user_id 필드 collation 은 utf8mb4_general_ci 로 설정 하였습니다. user_access 테이블 user_access_index 이라고 인덱스는 created_at, user_id 순으로 설정 하였습니다.

user_access 테이블 created_at 필드 조건절로 통해 users 테이블 하고 조인 후 SELECT 조회를 하게 된다면 동일한 데이터 타입 varchar 이기 떄문에 cast 는 발동하지 않을 것이고 인덱스도 순조롭게 잘 동작 될 것이라고 생각 할 수 있습니다. 하지만 서로 다른 collation 이 다르기 때문에

TABLE u 즉 users 테이블 조회시 user_access 테이블 통해 조회된 데이터 user_id 값을 이용해 users 테이블을 조회 하게 되는데 실행계획을 확인 해보면 type 이 ALL 즉 테이블 풀 스캔이 발생 됩니다. users 테이블 user_id 필드는 PK 이지만 문자열 컬럼의 문자셋/Collation 이 다르면 자동 형변환이 발생하고 그로 인해 인덱스를 무시하고 ALL 스캔을 수행합니다.

이것을 해결하기 위해서는

이렇게 user_access 테이블 user_id 필드를 utf8mb3_general_ci Collation 으로 변경 하면 문제를 해결 할 수 있습니다.

또는 이렇게 Collation 수정이 당장 어렵다면 명시적으로 쿼리 내에서 명시적 캐스팅을 통해 힌트를 줄 수 있습니다.

utf8mb4_0900_ai_ci Collation 에서 한글 비교 이슈

ai_ci 경우 악센트 및 대소문자 무시 한다고 설명 했었습니다. collation 테이블을 생성하고 ‘가’ ‘ㄱㅏ’ 를 INSERT 해서 WHERE 절을 통해 ‘가’ 로 조회를 하면 ‘ㄱㅏ’ 데이터도 포함 해서 조회 되는 것을 확인 할 수 있습니다.

하지만 초성, 중성, 종성 으로 구성되어 있는 ‘각’, ‘ㄱㅏㄱ’ ‘가ㄱ’ 데이터를 INSERT 하고 ‘각’ 으로 조회 하면 데이터 각 만 조회 되는데 왜 이렇게 조회가 되는지 한번 알아 보도록 하겠습니다.

먼저 utf8mb4-0900-aici collation 에서 완성형 문자인 가와 개별 문자인 ㄱ 그리고 ㅏ에 대해서 가중치 값이 어떻게 계산되는지 확인 해보겠습니다. DUCET 데이터로 위의 2개는 한글 초성 ㄱ, 중성 ㅏ에 대한 가중치 데이터 이고 아래 2개는 한글 문자 ㄱ, 한글 문자 ㅏ에 대한 가중치 데이터입니다.

DUCET 에서는 한글에 대해서 초성, 중성, 종성 별로 또 개별 문자 별로 가중치 데이터가 각각 다르게 존재 합니다. 이 데이터와 실제 MySQL 에서 WEIGHT_STRING 함수를 이용해서 계산된 가중치 값을 비교해 보겠습니다.

SQL 명령어 mysql> SELECT WEIGHT_STRING('가'); 대한 계산된 가중치 값을 볼 수 있는데 앞써 말했지만 한글의 경우 음절 문자 초성, 중성, 종성으로 분해 해서 가중치 값을 각각 계산하게 됩니다. 그래서 초성 ㄱ과 중성 ㅏ 데이터 사용 되고 이때 collation 민감도가 aici 이므로 첫번째 가중치 값만 사용 되기 때문에 각각 첫번째 가중치 값을 추출해 합쳐져 응답 (0x3BF53C73) 하는 것을 볼 수 있습니다.

SQL 명령어 mysql> SELECT WEIGHT_STRING('ㄱㅏ'); 확인 해보면 0x3BF53C73 으로 응답 하게 되는데요, 한글 초성 ㄱ과 한글 문자 ㄱ 한글 중성 ㅏ 하고 한글 문자 ㅏ 첫 번째 가중치 값이 모두 동일 하다 보니 결과적으로 각각 같은 가중치 값으로 응답 하는 것을 볼 수 있습니다.

이번에는 각 과 ㄱ, ㅏ, ㄱ에 대한 가중치 값을 비교 하면서 확인 해보겠습니다. 이 경우는 WEIGHT_STRING 함수를 통해 확인 해보면 최종 가중치 결과 값이 서로 다르게 나타나게 됩니다.

collation 민감도가 aici 이므로 음절 문자인 각 인 경우 초성, 중성, 종성 각각 가중치 데이터 값 중 첫번째 값만 사용 되어지고 한글 문자인 ㄱ, ㅏ, ㄱ 각각 가중치 값 첫번째로 사용 되어집니다.

이때 마찬가지로 최종 가중치 값을 확인 해보면 한글 종성 ㄱ 값 첫번째 가중치 값이 3CD1 이고 다른 한글 문자 ㄱ 경우는 첫번째 가중치 값이 3BF5 로 다르기 때문에 결과적으로 다른 가중치 값으로 표현하게 되어서 서로 다른 값으로 인식 하게 됩니다.

utf8mb4_0900_as_cs Collation 에서 한글 비교 이슈

앞써 케이스 경우는 utf8mb4-0900-ai-ci collation 에서는 첫 번째 가중치 값, 즉 primary 레벨의 값만 사용 되어졌는데 utf8mb4-0900-as-cs collation 경우는 세 번째 가중치 값까지 인 tertiary 까지 사용되어 집니다.

한글 초성과 한글 중성으로 구성되어 있는 가 경우는 tertiary 까지 사용되어 지기 때문에 각각의 가중치 데이터에서 같은 단계에 속하는 값들을 묶은 후 단계별 데이터들을 최종적으로 제로바이트로 연결한 값이 반환 되어 집니다.

한글 문자 ㄱ, ㅏ 경우를 보면 이것도 마찬가지로 collation 민감도가 as_cs 이기 때문에 세 번째 가중치 값까지 인 tertiary 까지 사용 되어져 최종 가중치 값이 0x3BF53C73000000200020000000040004 로 응답하게 됩니다.

서로 비교를 해보면 서로 다른 결과가 나타나게 되는데요, 이는 이전에 첫번째 가중치값만 비교했던 것과 다르게 세 번째 가중치 값까지인 tertiary 까지 비교 했기 때문에 서로 다른 값으로 나타나게 됩니다.

이번에는 한글 초성과 한글 중성 그리고 한글 종성 으로 구성되어 있는 각 값과 한글 문자 ㄱ, ㅏ, ㄱ 값을 비교 해보도록 하겠습니다. 이 경우도 collation 민감도가 as_cs 이기 때문에 세 번째 가중치 값까지 인 tertiary 까지 사용 되어져 있기 때문에, 가중치 값 데이터를 조합해서 각각 비교를 해보면 서로 다른 값으로 응답하기 때문에 결국은 다른 값으로 인식하게 됩니다.

utf8mb4_0900_ai_ci Collation 에서 한글 비교 이슈 해결법은?

여러가지 방법이 있겠지만 개인적으로 추천 하는 방법은 쿼리문을 보면 WHERE에 비교 조건에 utf8mb4_unicode_520_ci 추가해서 해당 collation 으로 비교 하면 ㄱㅏ 데이터 값은 응답값에 제외 된 것을 확인 할 수 있습니다. 이는 unicode_520_ci 경우는 가와 ㄱㅏ 를 서로 다른 문자로 인식 하기 때문에 쿼리 결과에서는 ㄱㅏ 데이터가 제외 된 것을 확인 할 수 있습니다.
